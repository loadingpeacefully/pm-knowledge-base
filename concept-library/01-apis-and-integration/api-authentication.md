## The world before this existed

A student management team shipped a new API endpoint to serve data to their mobile app — student name, grade, class balance, parent phone number, parent email. It worked perfectly. The app loaded fast, data was accurate, parents were happy.

Three months later, a security researcher sent an email. The endpoint was public. Anyone with the URL could pass in a student ID number and retrieve a parent's full contact details. No login, no password, no credential of any kind. Just: `GET /v1/student/details?studentId=1042`. Student 1042's parent phone number came back in the JSON response. So did 1043's. And 10,000 more.

The team had built the right data endpoint. They had tested that the data was correct. They had shipped on time. What they had not done: specify who was allowed to call it.

API authentication is the answer to that question. Before any data is touched, before any logic runs, the server asks: who are you, and can I verify that? If the answer is satisfactory, the request proceeds. If not, it stops immediately.

## What it is

API authentication is the mechanism that verifies a request comes from who it claims to be before any data or action is accessed.

Think of a hotel key card system. The hotel doesn't rely on people saying "I'm in room 412." It issues each guest a physical card encoded with their room assignment. When they approach the door, the lock reads the card — not the person's claim. A valid card opens the door. An invalid or expired card is rejected before the guest ever touches the handle.

In software: your API key, login token, or access credential is the key card. The server's authentication layer is the lock. Your word alone gets you nothing.

Three methods a PM will encounter:

**API keys** — a long unique string generated by the server, sent in a request header. Simple to issue and revoke. Used primarily for server-to-server calls (your system calling an external service). Risk: they don't expire automatically and can be accidentally leaked in code, error logs, or screenshots.

**JWTs (JSON Web Tokens)** — a signed digital token issued after login, containing the user's identity and an expiry time. The server validates the cryptographic signature — no database lookup needed. Tradeoff: once issued, a JWT cannot be cancelled before it expires.

**OAuth2** — a standard protocol where a user grants a third-party application limited, explicitly scoped access to their account. Instead of sharing a password, the user approves specific permissions. This is the "Sign in with Google" flow that shows you exactly what the app is requesting access to.

## When you'll encounter this as a PM

**Speccing a new endpoint.** Every API spec should answer: who is allowed to call this, and how do they prove it? An endpoint shipped without a declared auth requirement ships as a potential data exposure. This is a PM spec decision — not something to leave implied.

**Evaluating a third-party vendor API.** When integrating an external service, ask what auth method they use and whether their tokens are scoped. A read-only API token does far less damage if compromised than an admin-level token. The auth design is part of the vendor evaluation.

**A security incident happens.** "An API key was leaked in a GitHub commit" is one of the most common real incidents. Knowing the difference between rotating an API key (immediate fix, revoke and reissue) versus a JWT token compromise (can't revoke; must wait for expiry or force logout) tells you the severity and the right remediation timeline.

**Mobile and web have different auth rules.** Where a token is stored matters. Tokens stored in browser localStorage can be stolen by JavaScript — including malicious scripts from a compromised ad. Tokens in httpOnly cookies (set by the server, invisible to JavaScript) are not. Your engineers will make this call — you need to understand enough to ask whether it was made intentionally.

**A partner wants API access.** When you expose an API to a partner, you need to decide: API key (simple, revocable) or OAuth2 (user-consented, scoped)? The right answer depends on whether a human user is in the loop or it's a machine-to-machine integration. This is a PM product decision with security implications.

## How it actually works

Authentication happens before anything else in the request lifecycle. Understanding the sequence matters because it determines what your engineers build, what fails when it fails, and what your monitoring should track.

1. **The client sends a request with a credential.** API key in an `X-API-Key` header. JWT in an `Authorization: Bearer <token>` header. OAuth2 access token in the same Bearer format. The credential is always in the header — never in the URL, where it can be logged in server access logs.

2. **Auth middleware intercepts the request first.** Before route handlers, before business logic, before any database query. The middleware extracts the credential from the header.

3. **API key validation:** the server queries its key store — is this key in the database, is it active, which tenant or account does it belong to? This involves a database lookup on every request. Most implementations cache recent lookups to reduce latency.

4. **JWT validation:** the server re-computes the cryptographic signature using its secret key and compares against the token's embedded signature. If they match, the token is valid. No database lookup — this is the JWT's core advantage for high-throughput APIs. The server also checks the `exp` field (expiry timestamp). An expired signature that's otherwise valid returns `401`.

5. **OAuth2 token validation:** the server validates the access token (usually a short-lived opaque token or JWT) and reads the `scope` field. Scopes are explicit: `read:user`, `write:repos`, `admin:org`. An operation outside the token's scope returns `403 Forbidden`, not `401` — the caller is authenticated, but not authorized for this action.

6. **Authentication fails → 401 stops everything.** No data accessed. No business logic executed. The `401 Unauthorized` response is returned with a `WWW-Authenticate` header telling the client what credential type is expected. No further processing.

7. **Authentication succeeds → request proceeds with identity attached.** The middleware attaches the verified user ID, role, or tenant to the request context. Every subsequent layer — authorization checks, business logic, audit logging — reads from this context. This is how "who is making this request?" is answered in every part of the system.

8. **Token refresh (JWT-specific).** JWTs expire. Access tokens are typically short-lived: 15 minutes to 1 hour. When a token nears expiry, the client uses a separate **refresh token** — a longer-lived credential stored more securely, used only to obtain new access tokens — to silently get a fresh JWT without requiring the user to log in again. The critical property: refresh tokens can be revoked in the database (unlike JWTs). So "revoke this session" means invalidating the refresh token; the access JWT runs out naturally.

**The auth-authorization distinction:** authentication is identity verification ("you are who you say you are"). Authorization is access control ("you're allowed to do this"). A `401` means authentication failed — the server doesn't know who you are. A `403` means authentication passed but the caller lacks permission. Confusing these in error handling is a common and frustrating bug.

## The decisions this forces

**API keys vs JWTs — pick by use case, not preference.** API keys belong in server-to-server calls where a human user is never in the loop: your backend calling Stripe's API, a data pipeline calling your internal analytics endpoint. JWTs belong in user-facing flows where the token represents a logged-in user's session. The mistake: using API keys for user authentication (they're stateless but not user-scoped) or JWTs for server-to-server calls (overcomplicated for no benefit). Default: API keys for machine-to-machine, JWTs for user sessions. Never support both methods on the same endpoint without a documented reason for each — it doubles the attack surface and the test matrix.

**Token storage: localStorage vs httpOnly cookies.** localStorage is accessible to any JavaScript running on the page. An XSS vulnerability — a single injected script — can read all tokens from localStorage and exfiltrate them silently. httpOnly cookies are set by the server with a flag that makes them invisible to JavaScript entirely; they're only sent automatically with requests to the same domain. The tradeoff: httpOnly cookies introduce CSRF risk (cross-site request forgery — a malicious site can trigger requests that include the cookie). The mitigation is a CSRF token, which adds implementation complexity. For financial or sensitive data, httpOnly cookies are worth it. For a content-only app with low sensitivity, localStorage with short-lived JWTs is an acceptable tradeoff. This is a PM call because it's a product risk tolerance decision.

**Token expiry: short vs long.** A JWT stolen from a user's browser is valid until it expires. A 15-minute expiry limits the damage window but requires silent refresh logic that adds complexity. A 30-day expiry is simpler but dangerous if stolen. The right answer is threat-model specific: a payments endpoint should use short-lived tokens (15–30 minutes) with refresh. A public content feed where data isn't sensitive can tolerate longer-lived tokens. The engineer will default to whatever's easiest to implement — the PM should force the conversation by asking "what happens if this token is stolen?"

**OAuth2 scopes: minimum viable access.** When your product requests OAuth2 access to a user's account on another service, request only the scopes you actually use. Requesting `admin:org` when you only need `read:user` creates a large blast radius if your token is compromised — and causes user hesitation at the authorization consent screen. Scope creep is a product decision: each scope you add is a security surface and a conversion risk. The right level is exactly what the feature requires, reviewed annually as features evolve.

**Refresh token storage.** Refresh tokens are the most sensitive credential in the JWT flow — they're long-lived and can generate new access tokens indefinitely until revoked. Where you store them matters: memory (lost on page reload), localStorage (XSS risk), httpOnly cookies (safest, with CSRF mitigation). For high-sensitivity applications, refresh tokens go in httpOnly cookies. For most applications, a balance exists between security and implementation complexity — but the PM should ensure this was an explicit choice, not a default.

## Questions to ask your engineer

1. **"Which endpoints currently have no auth middleware — can we audit the full list?"** This reveals whether auth coverage is intentional or assumed. The answer "we don't have a central audit capability for this" is itself a flag. Expected: a list, or a tool that generates one. Concerning: "we think they all have auth."

2. **"Where are JWTs currently stored on the client — localStorage or httpOnly cookies — and was that a deliberate security decision?"** Engineers often default to localStorage because it's simpler. If the answer is "localStorage because it was easier," that's a known risk that was deferred, not a security decision. Understand what sensitivity lives behind these tokens.

3. **"When a user's JWT expires mid-session, what happens exactly — do we have silent refresh, or do we log them out?"** Silent refresh is better UX but more complex. If the team hasn't built it, users are getting logged out of half-completed actions. This reveals both a UX gap and whether the team has thought through the full auth lifecycle.

4. **"If an API key is leaked today — committed to a public GitHub repo — what's the rotation process and how many dependent services need updating?"** This surfaces the operational reality of API key management. If the answer involves manually updating config in 12 places, that's a real incident response problem. Expected: key revocation is a one-step operation, dependent services pick up the new key from a secrets manager automatically.

5. **"For our OAuth2 integrations with external services — what scopes did we request, and do we actually use all of them?"** Scope creep is common: scopes get added during development and never pruned. Unused scopes are pure downside — blast radius with no benefit. This question also tests whether anyone has audited this recently.

6. **"If auth middleware throws an unexpected error — an exception, a timeout on the key lookup — does the request fail open or fail closed?"** Fail open means the request passes through without authentication if the auth system has an error. This is a critical security bug disguised as a reliability decision. The correct answer is always fail closed: an auth error blocks the request. If the team hasn't tested this explicitly, it's an unknown.

7. **"On our most sensitive endpoints, do we have per-user rate limiting in addition to authentication?"** Authentication verifies identity; it doesn't prevent an authenticated user from hammering an endpoint. A legitimate user with a valid JWT can make 10,000 requests per minute. Rate limiting is a separate layer — covered in lesson 01.04. The question here tests whether the team thinks about authenticated abuse, not just anonymous abuse.

8. **"When we issue API keys to partners, do we have an automated rotation mechanism or are they permanent until manually revoked?"** Permanent API keys that never rotate are a long-term liability. Each key is a standing credential that works indefinitely if leaked. Key rotation policy (automatic expiry, scheduled rotation, rotation on personnel change) is an ops decision with security implications — the PM should ensure it exists.

## Real product examples — named, specific, with numbers

**Stripe: two-key architecture as threat model design.** Stripe issues two distinct API key types for every account: a publishable key (safe to embed in frontend JavaScript — it can only initiate payment flows, not access account data) and a secret key (server-only — full account access including refunds, payouts, and customer data). A leaked publishable key has minimal impact. A leaked secret key is a full account compromise. Stripe's two-key design makes the threat model explicit and enforces the right deployment pattern by design, not by documentation. The distinction has prevented countless incidents where developers accidentally committed credentials to public repositories. Stripe processes hundreds of billions of dollars annually — this design choice is directly load-bearing for trust.

**GitHub's OAuth2 scope system.** GitHub's developer OAuth implementation is the industry reference for granular scopes. Apps declare exactly what they need: `repo:read` (read code), `user:email` (read email address), `admin:org` (manage organization). Users see the scope list before granting access. Apps that request `admin:org` when they only need `repo:read` have visibly higher abandonment at the OAuth consent screen. The consequence: legitimate GitHub apps that over-scope get fewer authorizations. The scope system aligns developer incentives — minimum scope means better conversion — with user security. In 2023, GitHub introduced fine-grained personal access tokens with even more granular permissions, moving from coarse resource-type scopes to specific repository and action-level permissions.

**Zoom's API key to OAuth2 migration.** Zoom historically offered both JWT-based API keys (deprecated in 2023) and OAuth2 apps. Teams on the API key path — common in B2B integrations built quickly — hit a forcing function: Zoom deprecated JWT app types with a hard cutoff, requiring migration to OAuth2. Products that had not built OAuth2 flows faced a 6-month migration under deadline. The lesson for PMs: dependency on a vendor's auth mechanism is a product risk. When a vendor deprecates an auth method, you migrate on their timeline. The teams with clean OAuth2 integration had no incident. Teams using deprecated JWT keys scrambled.

**Firebase Authentication: managed auth as a build-vs-buy decision.** Firebase Auth handles the entire authentication stack — email/password, OAuth social logins, phone SMS verification, JWT issuance, token refresh — as a managed service. A startup can launch with production-grade auth in a day instead of building it in-house over weeks. The tradeoff: Firebase Auth is a Google service, and every authenticated user's session is managed on Google's infrastructure. For regulated industries (healthcare, finance), this raises data residency and vendor dependency questions. The PM decision: managed auth services like Firebase, Auth0, or Clerk compress time-to-market dramatically and handle security complexity correctly — but they're infrastructure dependencies that need to be scoped against compliance requirements before adoption, not after.

## What breaks and why

**The "no auth" endpoint pattern that compounds.** The most common and most expensive auth failure isn't a broken implementation — it's an endpoint that ships without any authentication layer because it was "internal only" or "for testing." These accumulate. Internal services call them without credentials because they never needed to set up credentials. Then infrastructure changes: a VPC misconfiguration, a new microservice boundary, a cloud migration. What was never accessible externally becomes accessible. The original exposure window can be months long before anyone checks. The PM prevention role: every new endpoint requires explicit auth documentation in the spec. "Internal only" is not a substitute for an auth decision. If the team can't say what auth method an endpoint uses, it doesn't ship.

**Token storage as deferred security debt.** The pattern from the KB source is canonical: tokens stored in plain localStorage to ship faster, with a "we'll migrate to httpOnly cookies in Phase 7" note in the technical debt backlog. Phase 7 rarely happens until an incident forces it. The migration from localStorage to httpOnly cookies is non-trivial: it requires coordination between frontend and backend, changes to every authenticated request flow, and regression testing of the full auth lifecycle. The cost of doing it right at the start is a few days. The cost of migrating under incident pressure is weeks plus reputational damage. PMs who approve "we'll harden auth later" are making an implicit risk calculation — they should make it explicit and document it, or push back.

**Refresh token revocation gaps.** JWT access tokens can't be revoked before expiry — this is structural to the design. The workaround is the refresh token: revoke the refresh token, and no new access tokens can be generated. But revocation requires a token store that gets checked on every refresh. Teams that implement JWT access tokens without a corresponding refresh token revocation mechanism have built a system where "log this user out on all devices" is impossible. For consumer products, this matters at scale: account takeover, abuse, compromised credentials. The PM role: ensure "force logout" is a first-class feature, tested in staging, before launch — not a post-incident retrofit.

**Scope drift in partner integrations.** OAuth2 scopes are requested at integration time and rarely revisited. A partner integration built two years ago with `admin:write` scope may have needed it then for a specific feature that has since been replaced by a dedicated API endpoint. The excess scope persists indefinitely. At scale, broad unused scopes in partner integrations represent a large attack surface: a compromised partner token can do far more than the current integration requires. PM responsibility: quarterly auth audit includes reviewing active partner token scopes against current integration functionality. Reducing scope is a security win with zero user impact.

## How this connects to the bigger system

Authentication is the entry gate for everything else in the curriculum. Understanding it compounds rapidly.

**What is an API (01.01)** establishes the request/response model that authentication sits on top of. The HTTP request lifecycle — auth middleware fires before business logic — only makes sense after you understand what an API call is. Authentication is not a separate concept; it's a mandatory layer in every API that handles non-public data.

**Webhooks vs Polling (01.03)** introduces outbound authentication: when your server calls an external webhook, how does the receiving endpoint know the request came from you and not a spoofed source? Webhook signature verification — sending a hash of the payload signed with a shared secret — is the auth pattern for outbound push notifications. The same core principle (prove identity via cryptographic signature) applies in a different direction.

**Rate Limiting (01.04)** is the layer after authentication in the security stack. Authentication answers "who are you?" Rate limiting answers "how much can you do?" Authenticated abuse — a legitimate user overloading an endpoint — requires rate limiting as the mitigation; revocation of their auth doesn't help if they just create a new account. The two concepts are additive, not substitutes.

**AI Agent Patterns (04.09)** creates a new auth problem that doesn't have settled answers yet. An AI agent acting on behalf of a user needs credentials to call APIs on the user's behalf. Do you issue long-lived API keys that the agent holds? Short-lived delegated tokens? OAuth2 scopes for agent actions? The emerging pattern is scoped delegation tokens with explicit user consent per action type — but implementation varies widely. This is an active design space where understanding classic auth patterns is required to evaluate new proposals critically.

## What senior PMs debate

**Passkeys will make passwords and JWTs obsolete for consumer auth — the timeline is the debate.** Passkeys (based on the WebAuthn standard, shipped by Apple, Google, and Microsoft natively since 2022–2023) replace passwords with cryptographic device-bound credentials. They're phishing-resistant by design, require no password to steal, and the UX is biometric unlock. Early adoption data (Google: 900M+ passkey uses in the year after launch; PayPal: 10% lower account takeover rate after rollout) shows the security gains are real. The debate is timeline and fragmentation: enterprises with legacy SSO infrastructure, regulated industries with specific auth compliance requirements, and users with older devices face real migration friction. For B2C consumer products, the answer is "implement passkeys now, keep password fallback." For B2B enterprise, it depends on whether your buyers control their SSO stack.

**The machine-to-machine auth problem is getting worse, not better.** Human user auth has passkeys and OAuth2. Machine-to-machine auth — service-to-service calls inside a microservices architecture — still largely relies on API keys or service account credentials that are stored in config files, secrets managers, or environment variables. The proliferation of microservices means more credentials, more rotation surface, more blast radius when something leaks. SPIFFE/SPIRE (cryptographic service identity for workloads) and short-lived Workload Identity tokens from cloud providers are the emerging answer, but adoption outside large engineering organizations is limited. PMs building platform products should be asking their infrastructure teams about the service identity model now, not when a service account credential is compromised.

**AI API authentication creates a new principal type with unclear governance.** An AI agent calling an API on behalf of a user is not the user and is not a traditional service account. It has a risk profile that doesn't fit the existing auth model: it may take actions the user didn't explicitly authorize in this specific session, it may be running in contexts the user can't observe, and it may compound actions across many systems. The question of how to scope, limit, and audit AI agent credentials is genuinely unsettled. The first generation of AI agents uses user-level OAuth2 tokens with broad scopes. The second generation will need purpose-bound, time-limited, action-scoped delegation tokens that didn't exist as a standard two years ago. PMs building AI-powered features that call external APIs should be designing for auditability and scope limitation from the start — the regulatory pressure to explain "what did your AI agent do with access to my account" is coming.
