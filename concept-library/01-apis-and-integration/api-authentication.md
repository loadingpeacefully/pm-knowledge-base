## The world before this existed

A student management team shipped a new API endpoint to serve data to their mobile app — student name, grade, class balance, parent phone number, parent email. It worked perfectly. The app loaded fast, data was accurate, parents were happy.

Three months later, a security researcher sent an email. The endpoint was public. Anyone with the URL could pass in a student ID number and retrieve a parent's full contact details. No login, no password, no credential of any kind. Just: `GET /v1/student/details?studentId=1042`. Student 1042's parent phone number came back in the JSON response. So did 1043's. And 10,000 more.

The team had built the right data endpoint. They had tested that the data was correct. They had shipped on time. What they had not done: specify who was allowed to call it.

API authentication is the answer to that question. Before any data is touched, before any logic runs, the server asks: who are you, and can I verify that? If the answer is satisfactory, the request proceeds. If not, it stops immediately.

## What it is

> **API Authentication:** The mechanism that verifies a request comes from who it claims to be before any data or action is accessed.

### The Hotel Key Card Analogy

The hotel doesn't rely on people saying "I'm in room 412." It issues each guest a physical card encoded with their room assignment. When they approach the door, the lock reads the card — not the person's claim.

- Valid card → door opens
- Invalid or expired card → rejected before the guest touches the handle

**In software:** Your API key, login token, or access credential is the key card. The server's authentication layer is the lock. Your word alone gets you nothing.

---

### Three Authentication Methods

| Method | How it works | Best for | Key risk |
|--------|-------------|----------|----------|
| **API Keys** | Long unique string generated by server, sent in request header | Server-to-server calls (your system calling external services) | ⚠️ No automatic expiry; easily leaked in code, error logs, or screenshots |
| **JWTs** | Signed digital token issued after login, containing user identity + expiry time. Server validates cryptographic signature—no database lookup needed | Stateless authentication, mobile apps, microservices | ⚠️ Cannot be cancelled before expiry once issued |
| **OAuth2** | User grants third-party app limited, explicitly scoped access to their account. User approves specific permissions instead of sharing password | Third-party app integrations, "Sign in with Google" flows | Shows user exactly what permissions the app is requesting |

## When you'll encounter this as a PM

### Speccing a new endpoint
Every API spec should answer: **who is allowed to call this, and how do they prove it?** An endpoint shipped without a declared auth requirement ships as a potential data exposure. This is a PM spec decision — not something to leave implied.

### Evaluating a third-party vendor API
When integrating an external service, ask:
- What auth method do they use?
- Are their tokens scoped?

⚠️ **Risk:** A read-only API token does far less damage if compromised than an admin-level token. Auth design is part of vendor evaluation.

### A security incident happens
"An API key was leaked in a GitHub commit" is one of the most common real incidents.

| Scenario | Response | Timeline |
|----------|----------|----------|
| **API key compromised** | Rotate key (revoke and reissue) | Immediate |
| **JWT token compromised** | Can't revoke; must wait for expiry or force logout | Depends on token TTL |

Knowing the difference tells you severity and the right remediation timeline.

### Mobile and web have different auth rules
Where a token is stored matters.

| Storage Method | Risk | JavaScript Access |
|---|---|---|
| Browser localStorage | Can be stolen by malicious scripts (including compromised ads) | ✓ Visible to JS |
| httpOnly cookies | Server-set, invisible to JavaScript | ✗ Protected from JS |

Your engineers will make this call — you need to understand enough to ask whether it was made intentionally.

### A partner wants API access
Decide between two auth models:

| Model | Best for | Key trait |
|-------|----------|-----------|
| **API key** | Machine-to-machine | Simple, revocable |
| **OAuth2** | User-in-the-loop | User-consented, scoped |

The right answer depends on whether a human user is involved. This is a PM product decision with security implications.

## How it actually works

Authentication happens before anything else in the request lifecycle. Understanding the sequence matters because it determines what your engineers build, what fails when it fails, and what your monitoring should track.

### Request authentication sequence

| Step | Action | Key Detail |
|------|--------|-----------|
| 1 | Client sends credential in header | API key (`X-API-Key`), JWT (`Authorization: Bearer`), or OAuth2 access token. **Never in URL** — prevents logging in server access logs. |
| 2 | Auth middleware intercepts | Runs before route handlers, business logic, or database queries. Extracts credential from header. |
| 3 | API key validation | Server queries key store: is key active? which tenant owns it? Involves database lookup per request (usually cached). |
| 4 | JWT validation | Server recomputes cryptographic signature using secret key. Checks `exp` field for expiry. No database lookup needed. |
| 5 | OAuth2 validation | Server validates access token, reads `scope` field (e.g., `read:user`, `write:repos`, `admin:org`). |
| 6 | Authentication fails | Returns `401 Unauthorized` with `WWW-Authenticate` header. No data accessed. No business logic executed. |
| 7 | Authentication succeeds | Middleware attaches verified user ID, role, or tenant to request context. All downstream layers use this context. |

### Token refresh (JWT-specific)

JWTs expire. Access tokens are typically short-lived (15 minutes to 1 hour). When nearing expiry, the client uses a separate **refresh token** to obtain a fresh JWT without requiring user login.

**Critical difference:** Refresh tokens can be revoked in the database (unlike JWTs). So "revoke this session" means invalidating the refresh token; the access JWT naturally expires.

### Authentication vs. authorization

> **Authentication:** Identity verification — "you are who you say you are"

> **Authorization:** Access control — "you're allowed to do this"

| Response | Meaning | What Failed |
|----------|---------|-------------|
| `401 Unauthorized` | Authentication failed | Server doesn't know who you are |
| `403 Forbidden` | Authentication passed, authorization failed | Caller lacks permission for this action |

⚠️ **Common bug:** Confusing `401` and `403` in error handling. Ensure your engineers distinguish between identity verification failure and permission denial.

## The decisions this forces

| Decision | Use Case | Tradeoff | Default |
|----------|----------|----------|---------|
| **API keys vs JWTs** | API keys: server-to-server calls (backend → Stripe, pipeline → analytics) | API keys: stateless but not user-scoped; JWTs: overcomplicated for machine calls | API keys for machine-to-machine, JWTs for user sessions |
| | JWTs: user-facing flows (logged-in sessions) | Never support both on same endpoint without documented reason — doubles attack surface & test matrix | |

⚠️ **Security risk:** Using API keys for user authentication or JWTs for server-to-server calls introduces unnecessary vulnerability.

---

| Decision | Method | Risk | Best For |
|----------|--------|------|----------|
| **Token storage** | localStorage | XSS vulnerability exposes all tokens silently | Content-only apps, low sensitivity |
| | httpOnly cookies | CSRF risk requires mitigation token; adds complexity | Financial/sensitive data |

> **XSS (Cross-Site Scripting):** Injected malicious script that reads tokens from localStorage and exfiltrates them.

> **CSRF (Cross-Site Request Forgery):** Malicious site triggers requests that automatically include httpOnly cookies.

**PM consideration:** This is a product risk tolerance decision.

---

| Decision | Duration | Damage Window | Complexity | Best For |
|----------|----------|----------------|-----------|----------|
| **Token expiry** | 15–30 minutes | Limited | Requires silent refresh logic | Payments endpoints, sensitive data |
| | 30+ days | Large if stolen | Simpler to implement | Public content feeds, low sensitivity |

⚠️ **Forcing function:** Ask engineers: "What happens if this token is stolen?" This drives threat-model-specific decisions.

---

### OAuth2 Scopes: Minimum Viable Access

**Principle:** Request only the scopes you actually use.

| Scope Level | Impact | Risk |
|-------------|--------|------|
| Request `admin:org` when only `read:user` needed | Large blast radius if compromised | User hesitation at consent screen |
| Request minimum required scope | Reduced blast radius | Lower conversion friction |

> **Scope creep:** Product decision to add each scope. Each addition increases security surface and conversion risk.

**PM ownership:** Review scopes annually as features evolve. Ensure alignment between requested access and actual feature requirements.

---

### Refresh Token Storage

> **Refresh tokens:** Long-lived credentials that generate new access tokens indefinitely until revoked. The most sensitive credential in JWT flows.

| Storage Method | Security | Usability | Best Practice |
|---|---|---|---|
| Memory | Lost on page reload | Session-only | Least secure |
| localStorage | XSS vulnerability | Persists across sessions | Not recommended for sensitive apps |
| httpOnly cookies + CSRF mitigation | Highest security | Full persistence | Recommended for high-sensitivity applications |

⚠️ **Verification required:** Ensure refresh token storage was an explicit security choice by the team, not a default implementation decision.

## Questions to ask your engineer

| Question | What this reveals | Red flag | Expected answer |
|----------|-------------------|----------|-----------------|
| **"Which endpoints currently have no auth middleware — can we audit the full list?"** | Whether auth coverage is intentional or assumed | "We think they all have auth" or "we don't have a central audit capability" | A list, or a tool that generates one |
| **"Where are JWTs currently stored on the client — localStorage or httpOnly cookies — and was that a deliberate security decision?"** | Whether token storage was a deliberate choice or a default | "localStorage because it was easier" | A documented security decision aligned with token sensitivity |
| **"When a user's JWT expires mid-session, what happens exactly — do we have silent refresh, or do we log them out?"** | Whether the team has thought through the full auth lifecycle and UX implications | No answer, or "we log them out" with no awareness of the UX gap | Silent refresh implemented, or a conscious choice to log out with mitigation |
| **"If an API key is leaked today — committed to a public GitHub repo — what's the rotation process and how many dependent services need updating?"** | The operational reality of API key management and incident response readiness | Manual updates required across multiple services | Key revocation is one-step; dependent services auto-fetch from secrets manager |
| **"For our OAuth2 integrations with external services — what scopes did we request, and do we actually use all of them?"** | Whether scope creep has accumulated unused permissions | "We're not sure" or unused scopes are in use | Scopes audited recently; only necessary permissions requested |
| **"If auth middleware throws an unexpected error — an exception, a timeout on the key lookup — does the request fail open or fail closed?"** | Whether the team understands the security implications of error handling | "Fail open for reliability" | Always fail closed; auth errors block requests |
| **"On our most sensitive endpoints, do we have per-user rate limiting in addition to authentication?"** | Whether the team thinks about authenticated abuse, not just anonymous abuse | No rate limiting on sensitive endpoints | Per-user rate limiting implemented on sensitive endpoints |
| **"When we issue API keys to partners, do we have an automated rotation mechanism or are they permanent until manually revoked?"** | Whether permanent credentials create long-term liability | Permanent API keys with no rotation policy | Automated rotation policy in place (expiry, scheduled rotation, or rotation on personnel change) |

### Why these questions matter

⚠️ **Auth coverage gaps** — Unapproved or undocumented endpoints without authentication are a common source of data breaches. A team that can't list them is flying blind.

⚠️ **Token storage risk** — localStorage tokens are readable by any JavaScript on the page. If a single dependency is compromised, all tokens are accessible. This is a known vulnerability, not a gray area.

⚠️ **Fail-open errors** — If auth middleware silently fails and lets requests through, you have a security hole that looks like a reliability choice. Test this explicitly.

⚠️ **API key rotation** — Permanent keys that are never rotated compound risk over time. Each key is a standing credential; the longer it lives, the higher the odds it's been compromised somewhere.

## Real product examples — named, specific, with numbers

### Stripe — Two-key architecture as threat model design

**What:** Stripe issues two distinct API key types:
- **Publishable key** — frontend-safe, limited to initiating payment flows only
- **Secret key** — server-only, full account access (refunds, payouts, customer data)

**Why:** A leaked publishable key has minimal impact. A leaked secret key is a full account compromise. The design makes the threat model explicit and enforces the right deployment pattern by design, not by documentation.

**Takeaway:** This architecture has prevented countless incidents where developers accidentally committed credentials to public repositories. Stripe processes hundreds of billions of dollars annually — this design choice is directly load-bearing for trust.

---

### GitHub — OAuth2 scope system

**What:** GitHub's developer OAuth uses granular, declarative scopes:
- `repo:read` — read code
- `user:email` — read email address
- `admin:org` — manage organization

Users see the full scope list before granting access.

**Why:** Apps requesting `admin:org` when they only need `repo:read` see visibly higher abandonment at the OAuth consent screen. The scope system aligns developer incentives (minimum scope = better conversion) with user security.

**Takeaway:** In 2023, GitHub introduced fine-grained personal access tokens with even more granular permissions — moving from coarse resource-type scopes to specific repository and action-level permissions.

---

### Zoom — API key to OAuth2 migration

**What:** Zoom deprecated JWT-based API keys in 2023, forcing migration to OAuth2. Teams using the legacy API key path faced a hard cutoff with a 6-month migration window.

**Why:** Dependency on a vendor's auth mechanism is a product risk. When a vendor deprecates an auth method, you migrate on their timeline, not yours.

**Takeaway:** Teams with clean OAuth2 integration had no incident. Teams using deprecated JWT keys scrambled under deadline.

*What this reveals:* Auth mechanism choices are infrastructure decisions with vendor-control consequences.

---

### Firebase Authentication — Managed auth as a build-vs-buy decision

**What:** Firebase Auth provides the entire authentication stack as a managed service:
- Email/password, OAuth social logins, phone SMS verification
- JWT issuance and token refresh
- Production-grade setup in ~1 day

**Why:** Startups can launch with production-grade auth in a day instead of building in-house over weeks. Firebase handles security complexity correctly.

**Tradeoff:** Every authenticated user's session is managed on Google's infrastructure, raising data residency and vendor dependency questions for regulated industries.

**Takeaway:** Managed auth services (Firebase, Auth0, Clerk) compress time-to-market dramatically — but they're infrastructure dependencies that must be scoped against compliance requirements *before* adoption, not after.

*What this reveals:* Build-vs-buy decisions require upfront compliance and regulatory review, not post-launch discovery.

## What breaks and why

### The "no auth" endpoint pattern that compounds

⚠️ **Critical risk:** Endpoints shipped without authentication layers accumulate silently and become exposed when infrastructure changes.

**How it happens:**
- Endpoint deployed as "internal only" or "for testing"
- No auth layer implemented
- Internal services call without credentials (never needed them)
- Infrastructure changes: VPC misconfiguration, microservice boundary shift, cloud migration
- What was never externally accessible suddenly becomes accessible
- Original exposure window: months before detection

**PM prevention:**
- Every new endpoint requires explicit auth documentation in spec
- "Internal only" ≠ an auth decision
- **Blocking criteria:** If the team can't name the auth method, the endpoint doesn't ship

---

### Token storage as deferred security debt

⚠️ **Risk pattern:** Shipping fast with localStorage, planning to migrate to httpOnly cookies "later"

| Decision | Cost at Start | Cost at Migration | Cost Under Incident |
|----------|---------------|-------------------|-------------------|
| Implement httpOnly cookies from day 1 | A few days | — | — |
| Use localStorage + plan migration | Days saved initially | Weeks of coordination | Weeks + reputational damage |

**What makes migration expensive:**
- Requires frontend-backend coordination
- Changes every authenticated request flow
- Full auth lifecycle regression testing needed
- Often trapped in "Phase 7" indefinitely

**PM responsibility:** Make risk calculations explicit. Document them. Or push back on "harden auth later" commitments.

---

### Refresh token revocation gaps

⚠️ **Structural vulnerability:** JWT access tokens cannot be revoked before expiry by design.

> **The workaround:** Revoke the refresh token → new access tokens cannot be generated

**The implementation gap:**
- Revocation requires a token store checked on every refresh
- Teams implementing JWT without refresh token revocation = no "force logout on all devices"
- At scale: account takeover, abuse, compromised credentials become uncontrollable

**PM responsibility:**
- "Force logout" is a first-class feature
- Test it in staging before launch
- Not a post-incident retrofit

---

### Scope drift in partner integrations

⚠️ **Attack surface risk:** OAuth2 scopes requested at integration time are rarely revisited.

**The accumulation pattern:**
- Partner integration built 2 years ago with `admin:write` scope
- Original feature requiring that scope has been replaced
- Excess scope persists indefinitely
- At scale: compromised partner token can do far more than integration actually needs

**PM responsibility:**
- Quarterly auth audit: compare active partner token scopes against current functionality
- Reducing scope = security win with zero user impact
- Make it a tracked task, not a "nice-to-have"

## How this connects to the bigger system

Authentication is the entry gate for everything else in the curriculum. Understanding it compounds rapidly.

| Concept | Connection | Why it matters |
|---------|-----------|-----------------|
| **What is an API (01.01)** | Establishes the request/response model that authentication sits on top of | HTTP request lifecycle — auth middleware fires before business logic — only makes sense after you understand what an API call is. Authentication is not a separate concept; it's a mandatory layer in every API that handles non-public data. |
| **Webhooks vs Polling (01.03)** | Introduces outbound authentication | When your server calls an external webhook, how does the receiving endpoint know the request came from you and not a spoofed source? Webhook signature verification — sending a hash of the payload signed with a shared secret — applies the same core principle (prove identity via cryptographic signature) in a different direction. |
| **Rate Limiting (01.04)** | The layer after authentication in the security stack | Authentication answers "who are you?" Rate limiting answers "how much can you do?" Authenticated abuse — a legitimate user overloading an endpoint — requires rate limiting as mitigation; revocation of their auth doesn't help if they just create a new account. The two concepts are additive, not substitutes. |
| **AI Agent Patterns (04.09)** | Creates a new auth problem without settled answers | An AI agent acting on behalf of a user needs credentials to call APIs on the user's behalf. Questions remain: long-lived API keys? Short-lived delegated tokens? OAuth2 scopes for agent actions? The emerging pattern is scoped delegation tokens with explicit user consent per action type — but implementation varies widely. This is an active design space where understanding classic auth patterns is required to evaluate new proposals critically. |

## What senior PMs debate

### Passkeys will make passwords and JWTs obsolete for consumer auth — the timeline is the debate

> **Passkeys:** Cryptographic device-bound credentials based on the WebAuthn standard, shipped natively by Apple, Google, and Microsoft since 2022–2023. Replace passwords with biometric unlock UX and are phishing-resistant by design.

**Evidence of impact:**
- Google: 900M+ passkey uses in the year after launch
- PayPal: 10% lower account takeover rate after rollout
- No password to steal; security gains are measurable

**The debate:**
| Factor | Challenge | Implication |
|--------|-----------|-------------|
| Legacy SSO | Enterprise infrastructure friction | B2B adoption slower |
| Regulatory compliance | Specific auth requirements in regulated industries | Mandatory fallback paths |
| Device age | Older devices lack passkey support | Migration window extends |
| User readiness | Biometric enrollment varies by region | Phased rollout required |

**PM action by context:**
- **B2C consumer products:** Implement passkeys now, keep password fallback
- **B2B enterprise:** Depends on whether buyers control their SSO stack

---

### The machine-to-machine auth problem is getting worse, not better

> **Machine-to-machine auth:** Service-to-service calls inside microservices architectures, currently relying on API keys or service account credentials stored in config files, secrets managers, or environment variables.

**Why this matters:**
- Microservices proliferation = more credentials
- More credentials = larger rotation surface
- Single credential leak = wider blast radius

**Emerging solutions:**
- **SPIFFE/SPIRE:** Cryptographic service identity for workloads
- **Workload Identity tokens:** Short-lived credentials from cloud providers (AWS, GCP, Azure)

⚠️ **Risk:** Adoption outside large engineering organizations remains limited. Credential compromise is a discovery trigger for many orgs, not a planning trigger.

**PM action:**
Ask your infrastructure teams about the service identity model *now* — not when a service account credential is compromised.

---

### AI API authentication creates a new principal type with unclear governance

> **AI principal:** An AI agent calling an API on behalf of a user — neither the user nor a traditional service account, with a distinct risk profile.

**Why the existing model breaks:**

| Risk Factor | Current Model | AI Agent Reality |
|------------|---|---|
| Authorization scope | User grants explicit consent per session | Agent may act on broad delegated authority |
| Observability | User can monitor their own actions | Agent runs in contexts user cannot observe |
| Compounding actions | Single transaction per decision | Agent compounds actions across many systems |
| Audit trail | Clear user intent | AI reasoning is opaque |

**The timeline:**

- **First generation:** User-level OAuth2 tokens with broad scopes (current)
- **Second generation:** Purpose-bound, time-limited, action-scoped delegation tokens (emerging)

⚠️ **Regulatory pressure incoming:** Expect demands to explain "What did your AI agent do with access to my account?"

**PM action:**
Build AI-powered features that call external APIs with auditability and scope limitation from the start. Design for the regulatory questions now.
